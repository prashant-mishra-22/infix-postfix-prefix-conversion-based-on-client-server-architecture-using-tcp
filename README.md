## ğŸ¯ Project Overview

This project implements a TCP-based client-server application for converting mathematical expressions between three different notations: **Infix**, **Prefix** (Polish Notation), and **Postfix** (Reverse Polish Notation). The system demonstrates practical implementation of socket programming, multi-threaded server architecture, and stack-based algorithms for expression parsing and conversion.

**Key Features**:
- **Six different conversion types** between infix, prefix, and postfix notations
- **Real-time validation** of input expressions on client side
- **Multi-client support** using threaded server architecture
- **Robust TCP communication** with proper message framing
- **Error handling** for invalid expressions and network issues

## ğŸ¯ Objectives

1. **Implement TCP-based client-server communication** for reliable data transfer
2. **Develop stack-based algorithms** for all six expression conversion types
3. **Create input validation mechanisms** for infix, prefix, and postfix expressions
4. **Design multi-threaded server** capable of handling multiple clients simultaneously
5. **Implement proper message framing** for TCP communication
6. **Document complete system architecture** and conversion algorithms

## ğŸ§  Expression Conversion Algorithms

### **Notation Types**
1. **Infix**: Operator between operands (A + B)
2. **Prefix**: Operator before operands (+ A B)  
3. **Postfix**: Operator after operands (A B +)

### **Algorithm Details**

#### **1. Infix to Postfix (Shunting Yard Algorithm)**
```python
Input: Infix expression with spaces
Output: Postfix expression

Algorithm:
1. Initialize empty stack and output list
2. Scan tokens left to right:
   - If operand: add to output
   - If '(': push to stack
   - If ')': pop from stack to output until '('
   - If operator: pop operators with â‰¥ precedence, then push current
3. Pop remaining operators to output
```

#### **2. Infix to Prefix**
```python
Steps:
1. Reverse infix expression
2. Swap '(' with ')' and vice versa
3. Convert to postfix using Shunting Yard
4. Reverse the result
```

#### **3. Prefix to Infix**
```python
Steps:
1. Read prefix expression from right to left
2. Push operands to stack
3. When operator encountered:
   - Pop two operands
   - Create "(operand1 operator operand2)"
   - Push result back to stack
```

#### **4. Postfix to Infix**
```python
Steps:
1. Read postfix expression left to right
2. Push operands to stack
3. When operator encountered:
   - Pop two operands
   - Create "(operand1 operator operand2)"
   - Push result back to stack
```

#### **5. Prefix to Postfix**
```python
Steps:
1. Read prefix expression from right to left
2. Push operands to stack
3. When operator encountered:
   - Pop two operands
   - Create "operand1 operand2 operator"
   - Push result back to stack
```

#### **6. Postfix to Prefix**
```python
Steps:
1. Read postfix expression left to right
2. Push operands to stack
3. When operator encountered:
   - Pop two operands
   - Create "operator operand1 operand2"
   - Push result back to stack
```

### **Operator Precedence**
```
Highest: * / (Precedence: 2)
Lowest:  + - (Precedence: 1)
```

### **TCP Communication Protocol**

#### **Message Format**
```
[Option Length][Option][Message Length][Message]
```
- **Option**: Conversion type (1-6)
- **Message**: Expression to convert
- **Length Headers**: Fixed size (2048 bytes) for reliable parsing

#### **Connection Flow**
1. Client connects to server on port 8000
2. Server accepts connection and creates thread
3. Client sends option and expression
4. Server processes request and sends response
5. Client displays result
6. Repeat until client disconnects

### **Validation Logic**

#### **Infix Validation**
```python
Rules:
1. Balanced parentheses
2. Operators must be between operands
3. No consecutive operators
4. Parentheses must contain valid expressions
```

#### **Postfix Validation**
```python
Rules:
1. Sufficient operands for each operator
2. Expression ends with operator
3. Valid operand count throughout evaluation
```

#### **Prefix Validation**
```python
Rules:
1. Expression starts with operator
2. Sufficient operands for each operator
3. Valid operand count when processed right-to-left
```

## ğŸš§ Challenges Faced

### **Technical Challenges**

1. **TCP Message Framing**
   - Issue: TCP is stream-oriented, messages can be fragmented
   - Solution: Implemented fixed-length headers for message boundaries

2. **Expression Validation**
   - Issue: Complex rules for different notation types
   - Solution: Created separate validation functions for each notation

3. **Multi-Client Synchronization**
   - Issue: Race conditions with shared resources
   - Solution: Used threading with separate client handlers

4. **Algorithm Implementation**
   - Issue: Converting between all six notation types
   - Solution: Used stack-based algorithms and derived some conversions from others

5. **Network Configuration**
   - Issue: Clients unable to connect across networks
   - Solution: Proper IP configuration and firewall settings

### **Algorithm Challenges**

1. **Infix to Prefix Conversion**
   - Challenge: Requires expression reversal and parentheses swapping
   - Solution: Implemented 4-step process using existing infix-to-postfix

2. **Operator Precedence**
   - Challenge: Handling different precedence levels in conversions
   - Solution: Created `precedence()` function with proper levels

3. **Parentheses Handling**
   - Challenge: Nested parentheses in complex expressions
   - Solution: Stack-based approach with proper push/pop logic

### **Development Challenges**

1. **Debugging Network Issues**
   - Challenge: Difficult to debug client-server communication
   - Solution: Added verbose logging on both sides

2. **Testing Multiple Scenarios**
   - Challenge: Testing all six conversions with various expressions
   - Solution: Created comprehensive test cases

3. **Documentation**
   - Challenge: Explaining complex algorithms clearly
   - Solution: Used step-by-step explanations with examples
